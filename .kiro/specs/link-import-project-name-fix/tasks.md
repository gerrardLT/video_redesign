# Implementation Plan

## Overview

本任务列表遵循探索式 bugfix 方法论：先在**未修复**代码上复现缺陷（任务 1，预期 FAIL）并固化要保留的基线行为（任务 2，预期 PASS），再实施修复（任务 3.1），最后复用同一测试验证"缺陷已修复"（任务 3.2）与"无回归"（任务 3.3），以检查点（任务 4）收尾。缺陷为：链接导入视频未填写项目名称时，下载完成后未把 yt-dlp 真实标题回写到 `project.name`，名称恒为占位符 `导入视频 - {platform}`。

## Tasks

- [ ] 1. 编写 Bug 条件探索测试（在修复前）
  - **Property 1: Bug Condition** - 未填写名称时未回写真实视频标题
  - **CRITICAL**: 本测试在未修复代码上必须 FAIL —— 失败即证明缺陷存在
  - **DO NOT** 在测试失败时尝试修复测试或代码（此阶段只做复现与理解）
  - **NOTE**: 本测试编码的是期望行为，修复后它通过即可验证修复（见任务 3.2）
  - **GOAL**: 暴露反例，证明缺陷确实存在
  - **Scoped PBT Approach**: 由于核心是确定性的名称归一化/回写决策逻辑，将属性聚焦到可复现的具体失败用例上：用 fast-check 生成任意非空白 yt-dlp 标题（含首尾空白、超长、Unicode）与任意平台，断言"下载完成回写决策产出的名称"等于 `resolveImportedProjectName(title, platform)` 的期望结果
  - 以 design.md「Bug Condition」与「Exploratory Bug Condition Checking」为准实现测试，断言对象为下载完成时 `project.update` 的 data（可对 `prisma.project.update` 调用参数打桩观测，或将回写决策抽为可测单元）
  - 测试断言须与 Expected Behavior Property 1 一致：
    - 非空白标题：`finalName = truncate(trim(ytdlpTitle), 100)` 且 `length ≤ 100`
    - 空/纯空白标题：`finalName` 为非空兜底 `导入视频 - {platform}`
  - 在**未修复**代码上运行（此时回写 data 不含 `name`，名称恒为占位符）
  - **EXPECTED OUTCOME**: 测试 FAILS（正确——证明缺陷存在）
  - 记录反例（例如：标题 `"夏日海边 vlog"` 导入抖音，期望 `夏日海边 vlog`，实际 `导入视频 - douyin`），用于理解根因
  - 测试编写、运行、失败被记录后，标记此任务完成
  - 文件名使用 `.property.test.ts`，运行于 Node 环境（Vitest 4 + fast-check）
  - _Requirements: 1.1, 1.2_

- [ ] 2. 编写保留性属性测试（在修复前）
  - **Property 2: Preservation** - 已填写名称时不被覆盖且下载流程不变
  - **IMPORTANT**: 遵循"观察优先（observation-first）"方法论
  - 先在**未修复**代码上观察非缺陷输入（`isBugCondition` 返回 false，即用户已填写有效名称）的实际行为：
    - 观察：`validateAndImport` 在用户填写名称时项目落用户名称，且下载完成回写决策不改动 `name`
    - 观察：无论是否回写名称，`videoUrl`/`status=PARSING` 始终被写入
    - 观察：下载失败走 FAILED 分支且不回写 `name`
  - 以 design.md「Preservation Requirements」与「Preservation Checking」为准，编写属性测试固化观察到的行为模式：
    - 用 fast-check 生成任意 yt-dlp 标题，固定 `nameAutoGenerated=false`，断言回写决策产出的 update data **不含** `name`（名称保持用户输入）
    - 断言 `videoUrl`/`status=PARSING` 字段恒被写入（验证回写名称未挤掉既有字段）
    - 断言失败路径不写 `name`
  - 属性测试可自动覆盖大量输入域，对"非缺陷输入行为不变"提供强保证
  - 在**未修复**代码上运行
  - **EXPECTED OUTCOME**: 测试 PASS（确认要保留的基线行为）
  - 测试编写、运行、在未修复代码上通过后，标记此任务完成
  - _Requirements: 3.1, 3.2, 3.3_

- [ ] 3. 修复：未填写名称时回写真实视频标题

  - [ ] 3.1 实现修复
    - 改动 1（`src/lib/video-import-service.ts`）：新增并导出纯函数 `resolveImportedProjectName(rawTitle, platform)`，按 `trim → 截断到 100 → 空则兜底为 导入视频 - {platform}` 归一化标题；新增常量 `PROJECT_NAME_MAX_LENGTH = 100`，必须先 trim 再截断，占位符与建项目逻辑复用同一构造方式
    - 改动 2（`src/lib/video-import-service.ts` 中 `validateAndImport`）：计算意图 `const nameAutoGenerated = isBlank(projectName)`；建项目仍按现有逻辑落占位符（保持 `name` 非空）；在 `videoDownloadQueue.add(...)` 的 job data 中追加 `nameAutoGenerated`
    - 改动 3（`src/workers/download-video.ts` 中 `processDownloadVideo` 步骤 4 的 `prisma.project.update`）：`DownloadVideoJobData` 增加 `nameAutoGenerated: boolean`；在已有的回写 `videoUrl`+`status=PARSING` 的同一次 `update` 上条件性追加 `name`：仅当 `job.data.nameAutoGenerated === true` 时写 `name = resolveImportedProjectName(title, platform)`，否则完全不触碰 `name`
    - 失败路径（catch 块）保持不变，不写 `name`；`downloadWithYtDlp` 下载逻辑不改动；复用现有 `title` 返回值与同一次 update，不新增数据库往返
    - 同步更新相关注释，确保与新逻辑一致（遵循代码质量规则）
    - _Bug_Condition: isBugCondition(X) = isBlank(X.projectName)（design.md）_
    - _Expected_Behavior: 非空白标题 finalName = truncate(trim(ytdlpTitle), 100)；空白标题 finalName = 非空兜底（design.md Property 1）_
    - _Preservation: 已填写名称不被覆盖；videoUrl/status/解析/上传/计费/FAILED 流程不变（design.md Preservation Requirements）_
    - _Requirements: 2.1, 2.2, 2.3, 3.1, 3.2, 3.3_

  - [ ] 3.2 验证 Bug 条件探索测试现在通过
    - **Property 1: Expected Behavior** - 未填写名称时回写真实视频标题
    - **IMPORTANT**: 重新运行任务 1 的**同一测试**——不要新写测试

    - 运行任务 1 的 Bug 条件探索测试

## Notes

- **Property 1 / Property 2**：标注 `**Property N:**` 的任务支持悬停查看测试状态。Property 1 为 Bug 条件（任务 1 探索 + 任务 3.2 修复后期望行为），Property 2 为保持性（任务 2 + 任务 3.3）。
- **observation-first**：保持性测试（任务 2）须先在未修复代码上观察真实输出再固化，不可凭假设编写。
- **不要新写验证测试**：任务 3.2 / 3.3 均为重跑任务 1 / 任务 2 中已有的同一测试，禁止另写新测试。
- 属性测试使用 Vitest 4 + fast-check，文件名带 `.property.test.ts`，运行于 Node 环境。
- 遵循项目规范：不使用 fallback / 静默降级 / 假数据；改动代码时同步更新注释。
