# 链接导入项目名称修复 Bugfix 设计

## Overview

通过"链接导入视频"功能导入视频时，若用户未填写项目名称，系统在 `validateAndImport` 创建项目时使用占位符 `导入视频 - {platform}`。下载 Worker（`download-video.ts`）通过 yt-dlp 在下载阶段已经拿到了视频真实标题（`title`），但下载完成后仅把 `videoUrl` 与 `status` 回写到 `project`，并未把真实标题回写到 `project.name`，导致项目名称永远停留在占位符上。

本次修复的核心思路：

1. **显式标记"项目名称是否为系统自动生成"**。在 `validateAndImport` 判定用户是否填写了项目名称（`isBlank(projectName)`），并把该意图（`nameAutoGenerated`）随下载任务一起传递给下载 Worker。这样下载完成时才能确定是否允许用真实标题覆盖名称。
2. **下载完成时按意图回写名称**。仅当 `nameAutoGenerated === true` 时，下载 Worker 才用 yt-dlp 标题经过 `trim → 截断(100) → 兜底` 处理后回写 `project.name`；否则完全不触碰 `name`。
3. **把名称归一化逻辑抽成纯函数** `resolveImportedProjectName(rawTitle, platform)`，集中处理 trim、超长截断、空标题兜底，便于用 fast-check 做属性测试。

修复是最小侵入式的：只在创建任务时附带一个布尔意图，在下载完成回写 `project` 的那一次 `update` 上条件性追加 `name` 字段，其余下载流程（解析、下载、上传 OSS、回写 `videoUrl`/`status`、触发解析、余额校验、失败标记 FAILED）保持不变。

## Glossary

- **Bug_Condition (C)**: 触发缺陷的条件 —— 用户通过链接导入视频时未填写项目名称（`isBlank(projectName)` 为 true）。
- **Property (P)**: 期望行为 —— 下载完成后，未填写名称的项目其 `name` 反映 yt-dlp 真实标题（经 trim/截断/兜底处理），而非占位符。
- **Preservation**: 必须保持不变的行为 —— 用户已显式填写名称时名称不被覆盖；以及解析、下载、上传 OSS、回写 `videoUrl`/`status`、触发解析、余额校验、失败标记 FAILED 等既有下载流程。
- **validateAndImport**: `src/lib/video-import-service.ts` 中的导入服务函数，负责验证链接、创建 `Project`（`status=DOWNLOADING`）、创建 `VideoDownloadTask`、入队下载任务。
- **processDownloadVideo / downloadWithYtDlp**: `src/workers/download-video.ts` 中的下载 Worker 主逻辑与 yt-dlp 下载函数，后者返回 `{ localPath, title }`。
- **nameAutoGenerated**: 新增的意图标记，表示项目名称是系统自动生成的占位符（用户未填写），允许下载完成后用真实标题覆盖。
- **resolveImportedProjectName**: 新增纯函数，把原始 yt-dlp 标题归一化为最终项目名称（trim → 截断到 100 → 空则兜底为占位符）。
- **project.name**: `Project` 模型的必填字段（`String`），项目展示名称，长度上限按业务约定为 100 字符。

## Bug Details

### Bug Condition

缺陷发生在：用户通过链接导入视频且**未填写项目名称**时。此时 `validateAndImport` 用占位符 `导入视频 - {platform}` 建项目；下载 Worker 在 `downloadWithYtDlp` 中已取得真实 `title`，但下载完成的 `prisma.project.update` 只写 `videoUrl` 与 `status`，从不写 `name`，导致名称恒为占位符。

**Formal Specification:**
```
FUNCTION isBugCondition(X)
  INPUT: X of type ImportInput   // { projectName?, sourceUrl, platform, ytdlpTitle }
  OUTPUT: boolean

  // 用户未提供项目名称（undefined、空字符串或纯空白）即触发缺陷
  RETURN isBlank(X.projectName)
END FUNCTION
```

其中 `isBlank(s)` 当 `s` 为 `undefined`、空字符串或仅含空白字符时返回 true。

### Examples

- **缺陷示例 1**：未填写名称导入抖音视频，yt-dlp 标题为 `"夏日海边 vlog"`。期望项目名为 `夏日海边 vlog`，实际为 `导入视频 - douyin`。
- **缺陷示例 2**：未填写名称导入快手视频，yt-dlp 标题为 `"  我的旅行  "`（含首尾空白）。期望项目名为 trim 后的 `我的旅行`，实际为 `导入视频 - kuaishou`。
- **缺陷示例 3**：未填写名称导入，yt-dlp 标题长度 250 字符。期望项目名为前 100 字符，实际为 `导入视频 - weixin`。
- **边界示例（空标题兜底）**：未填写名称导入，yt-dlp 标题为空字符串或纯空白。期望项目名为合理兜底 `导入视频 - {platform}`（非空），不能是空名称。
- **正常（非缺陷）示例**：用户填写了名称 `我的项目`。期望项目名始终为 `我的项目`，下载完成不被标题覆盖 —— 此输入不满足 bug 条件，行为须保持不变。

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors（必须保持不变）:**
- 用户已显式填写项目名称时，名称始终等于用户输入，下载完成后**不被**真实标题覆盖（对应需求 3.1）。
- 既有下载流程保持原样：链接验证与短链接提取、yt-dlp 解析+下载、上传 OSS、回写 `videoUrl`/`status=PARSING`、下载任务 `status=COMPLETED`、解析前余额预检、余额不足标记项目 FAILED、触发 `video-parse` 队列、进度回写（对应需求 3.2）。
- 视频下载失败时，下载任务与项目仍被标记为 FAILED 并记录错误信息，**不回写**项目名称；BullMQ 重试逻辑不变（对应需求 3.3）。

**Scope:**
所有不满足 bug 条件的输入（用户已填写名称）应完全不受本次修复影响，包括：
- 用户填写名称的链接导入项目的命名行为。
- 所有下载流程中与名称无关的步骤（下载、上传、解析、计费、状态流转）。
- 下载失败路径（FAILED 标记与错误信息记录）。

> 注：实际期望的"正确命名行为"在下方 Correctness Properties（Property 1）中定义；本节聚焦于**不得改变**的内容。

## Hypothesized Root Cause

基于代码分析，根因明确而非多种猜测：

1. **下载完成回写缺少 `name`（主因，已确认）**：`download-video.ts` 步骤 4 的 `prisma.project.update` 仅写入 `{ videoUrl, status: 'PARSING' }`，`downloadWithYtDlp` 返回的 `title` 在打印日志后被丢弃，从未参与 `project.name` 的更新。

2. **缺少"用户是否填写名称"的意图传递**：`validateAndImport` 在建项目时就用 `projectName || 占位符` 落库，但既没有把"是否为占位符/自动生成"的意图持久化，也没有传给下载 Worker。下载 Worker 无从判断该不该覆盖名称，因此即便要回写也缺乏决策依据。

3. **缺少标题归一化处理**：现有代码对 yt-dlp 标题没有 trim、超长截断、空标题兜底处理（仅在 `downloadWithYtDlp` 内对完全取不到时给了 `'导入视频'` 默认），不满足边界要求（空白/超长）。

## Correctness Properties

Property 1: Bug Condition - 未填写名称时回写真实视频标题

_For any_ 满足 bug 条件的输入（`isBugCondition` 返回 true，即用户未填写项目名称），修复后的完整导入+下载流程 `F'` 在下载成功完成后 SHALL 使项目名称反映 yt-dlp 真实标题，且满足边界处理：当标题非空白时，`project.name = truncate(trim(ytdlpTitle), 100)`；当标题为空或纯空白时，`project.name` 为非空的合理兜底名称（`导入视频 - {platform}`），不得为空。

**Validates: Requirements 2.1, 2.2, 2.3**

Property 2: Preservation - 已填写名称时与既有下载流程行为不变

_For any_ 不满足 bug 条件的输入（`isBugCondition` 返回 false，即用户已显式填写项目名称），修复后的流程 `F'` SHALL 产生与修复前 `F` 完全相同的结果：项目名称始终等于用户输入且不被标题覆盖，且解析/下载/上传 OSS/回写 `videoUrl`、`status`/触发解析/余额校验/失败标记 FAILED 等既有行为完全保持不变。

**Validates: Requirements 3.1, 3.2, 3.3**

## Fix Implementation

### Changes Required

假设根因分析正确，需要三处协同改动 + 一个纯函数。

**改动 1 —— 新增名称归一化纯函数**

**File**: `src/lib/video-import-service.ts`

新增并导出纯函数，集中处理 trim / 超长截断 / 空标题兜底，便于属性测试：

```
const PROJECT_NAME_MAX_LENGTH = 100

FUNCTION resolveImportedProjectName(rawTitle, platform)
  INPUT: rawTitle of type string | null | undefined, platform of type string
  OUTPUT: string (非空)

  trimmed ← trim(rawTitle ?? '')
  IF trimmed = '' THEN
    RETURN `导入视频 - ${platform}`          // 兜底：与建项目占位符一致，保证非空
  END IF
  RETURN trimmed.slice(0, PROJECT_NAME_MAX_LENGTH)  // 截断到 100 字符上限
END FUNCTION
```

要点：必须**先 trim 再截断**，与 bugfix.md 中 `truncate(trim(title), 100)` 的定义一致；截断后不再二次 trim。占位符字面量与建项目时保持一致，复用同一个常量/构造方式避免漂移。

**改动 2 —— 在导入服务中传递"是否自动生成名称"意图**

**File**: `src/lib/video-import-service.ts`
**Function**: `validateAndImport`

1. 计算意图：`const nameAutoGenerated = isBlank(projectName)`（`isBlank` 判定 undefined/空/纯空白）。
2. 建项目时仍按现有逻辑落占位符（保持 `name` 非空约束）：`name: projectName?.trim() || 导入视频占位符`。
3. 把意图随下载任务入队数据传给 Worker：在 `videoDownloadQueue.add(...)` 的 job data 中追加 `nameAutoGenerated`。

> 设计选择：意图通过 **BullMQ job data** 传递（`DownloadVideoJobData` 增加 `nameAutoGenerated: boolean`），不新增 Prisma 字段，避免数据库迁移。job data 由 BullMQ 持久化于 Redis，重试时复用同一数据，足以支撑下载完成时的决策。若后续需要在 DB 层可查询该意图，可作为备选方案在 `VideoDownloadTask` 增加 `nameAutoGenerated Boolean @default(false)`（属于 Prisma schema 变更，需单独确认）。

**改动 3 —— 下载完成时条件回写名称**

**File**: `src/workers/download-video.ts`
**Function**: `processDownloadVideo`（步骤 4 的 `prisma.project.update`）

1. `DownloadVideoJobData` 增加 `nameAutoGenerated: boolean`。
2. 在已有的"回写 `videoUrl` + `status=PARSING`"那一次 `update` 上，条件性追加 `name`：

```
const updateData = { videoUrl: ossUrl, status: 'PARSING' }
if (job.data.nameAutoGenerated) {
  updateData.name = resolveImportedProjectName(title, platform)
}
await prisma.project.update({ where: { id: projectId }, data: updateData })
```

要点：
- 仅当 `nameAutoGenerated === true` 才写 `name`，否则完全不触碰 `name`（保证 Preservation）。
- 复用现有这次 `update`，不新增数据库往返。
- 失败路径（catch 块）保持不变，**不**写 `name`。
- `title` 来自现有 `downloadWithYtDlp` 返回值，无需改动下载函数本身。

### 不改动的部分（显式声明）

- `downloadWithYtDlp` 的下载/反爬/无水印/错误翻译逻辑不变。
- `uploadToOSS`、余额预检（`estimateParseCreditCost`/`getBalance`）、余额不足标记 FAILED、触发 `video-parse`、进度回写、Worker 并发/限流配置均不变。
- 失败标记 FAILED 与错误信息记录、BullMQ 重试不变。

## Testing Strategy

### Validation Approach

采用两阶段策略：先在**未修复**代码上复现缺陷以确认根因（探索性 Bug 条件检查），再验证修复正确（Fix Checking）且既有行为不被破坏（Preservation Checking）。属性测试使用 Vitest 4 + fast-check，文件名带 `.property.test.ts`，运行于 Node 环境。

### Exploratory Bug Condition Checking

**Goal**: 在实现修复前先复现缺陷，确认/推翻根因。若推翻则需重新假设根因。

**Test Plan**: 针对名称归一化逻辑与回写决策编写测试，并在**未修复**代码上运行，观察项目名称在下载完成后仍停留在占位符。由于下载 Worker 依赖 yt-dlp/OSS/Redis 等外部资源，缺陷复现以"回写决策 + 名称归一化"为核心：对下载完成的 `project.update` 数据做断言（可对 `prisma.project.update` 调用参数打桩观测，或抽出回写决策为可测单元）。

**Test Cases**:
1. **真实标题回写缺失**：未填写名称 + yt-dlp 返回非空标题，断言下载完成后 `project.name` 等于标题归一化结果（在未修复代码上失败 —— 仍为占位符）。
2. **首尾空白未 trim**：标题 `"  我的旅行  "`，断言最终名称为 `我的旅行`（未修复失败）。
3. **超长未截断**：标题长度 > 100，断言最终名称长度为 100 且等于前 100 字符（未修复失败）。
4. **空标题兜底**（边界）：标题为空/纯空白，断言最终名称为非空兜底 `导入视频 - {platform}`（未修复时恰好为占位符，可能通过 —— 用于界定边界，非主复现点）。

**Expected Counterexamples**:
- 下载完成后 `project.update` 的 data 不含 `name`，项目名恒为 `导入视频 - {platform}`。
- 根因：步骤 4 回写遗漏 `name` + 缺少意图传递与标题归一化。

### Fix Checking

**Goal**: 验证所有满足 bug 条件的输入，修复后流程产出期望名称（含空/超长边界）。

**Pseudocode:**
```
FOR ALL X WHERE isBugCondition(X) DO        // 即 isBlank(X.projectName)
  result ← importAndDownload'(X)            // F'：修复后的导入+下载
  finalName ← result.project.name

  IF isBlank(X.ytdlpTitle) THEN
    ASSERT isBlank(finalName) = false        // 兜底非空
  ELSE
    ASSERT finalName = truncate(trim(X.ytdlpTitle), 100)
  END IF
END FOR
```

**实现方式**：以纯函数 `resolveImportedProjectName(rawTitle, platform)` 为 Fix Checking 的主载体，用 fast-check 生成任意 `rawTitle`（含空串、纯空白、含首尾空白、超长、含 Unicode）与任意 `platform`，断言：空白输入 → 返回非空兜底；非空白输入 → 返回 `trim` 后截断到 100 的结果且长度 ≤ 100。

### Preservation Checking

**Goal**: 验证所有不满足 bug 条件的输入（用户已填写名称），修复后行为与修复前完全一致。

**Pseudocode:**
```
FOR ALL X WHERE NOT isBugCondition(X) DO    // 即用户已填写有效名称
  ASSERT F(X) = F'(X)                        // 名称恒为用户输入；下载流程其余行为不变
END FOR
```

**Testing Approach**: 推荐属性测试，因为它能自动覆盖大量输入域、捕捉手写用例遗漏的边界，并对"非缺陷输入行为不变"提供强保证。

**Test Plan**: 先在未修复代码上观察"用户已填写名称"路径与下载完成回写行为，再写属性测试固化：当 `nameAutoGenerated === false` 时，下载完成的 `project.update` data 不包含 `name`（即名称不被改动）。

**Test Cases**:
1. **已填写名称不被覆盖**：`nameAutoGenerated=false` 时，对任意 yt-dlp 标题，回写决策产出的 update data 不含 `name`，项目名保持用户输入。
2. **回写决策幂等于流程其余字段**：无论是否回写 `name`，`videoUrl`/`status=PARSING` 始终被写入（验证回写名称未挤掉既有字段）。
3. **失败路径不回写名称**：模拟下载/上传抛错，断言走 FAILED 分支且未写 `name`。
4. **归一化纯函数对非空白输入的稳定性**：`resolveImportedProjectName` 对同一输入多次调用结果一致（纯函数无副作用）。

### Unit Tests

- `resolveImportedProjectName`：空串/纯空白 → 兜底 `导入视频 - {platform}`；普通标题 → 原样；首尾空白 → trim；> 100 字符 → 截断到 100；恰好 100 字符 → 不变。
- `validateAndImport`：未填写名称 → 入队 job data 含 `nameAutoGenerated=true` 且项目落占位符；填写名称（含含空白被 trim 后非空）→ `nameAutoGenerated=false` 且项目落用户名称。
- 下载完成回写决策：`nameAutoGenerated=true` 时 update data 含归一化后的 `name`；`false` 时不含 `name`。

### Property-Based Tests

- **Fix（Property 1）**：fast-check 生成任意标题（空/空白/含空白/超长/Unicode）与平台，断言 `resolveImportedProjectName` 输出非空且 `length ≤ 100`，非空白输入等于 `trim` 后取前 100 字符。
- **Preservation（Property 2）**：fast-check 生成任意标题，固定 `nameAutoGenerated=false`，断言回写决策产出的 update data 不含 `name`（名称不变），且 `videoUrl`/`status` 字段恒被写入。

### Integration Tests

- 未填写名称的完整链接导入流程（建项目 → 入队 → 下载完成回写），断言项目名最终为真实标题归一化结果（在可控/桩化外部依赖的集成环境中）。
- 已填写名称的完整流程，断言项目名全程等于用户输入。
- 下载失败流程，断言项目与任务为 FAILED、记录错误信息、未回写名称，且解析未被触发。
