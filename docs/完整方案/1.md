下面给你一版**生产级、可落地的技术方案**。我会基于你现有业务全景图里的流程：**视频导入 → 解析 → 编辑 → 单组生成 → 合并导出**，保留你已经做对的核心设计：**每个 Shot 独立成组、首帧锚定、短镜头合并、单组生成、最终 FFmpeg concat 导出**。

# 一、总体判断：这不是“AI 调接口”，而是一条视频生产流水线

你的业务本质是一个**异步媒体生产系统**，不是普通 CRUD。上线后最容易出问题的不是页面，而是：

1. 视频下载失败、上传中断、格式不兼容。
2. FFmpeg 抽帧不稳定，镜头切分过多或过少。
3. Gemini 输出 JSON 不稳定，时间戳/分镜边界有偏差。
4. Seedance 生成任务慢、失败、限流、结果 URL 过期。
5. 用户重复点击导致重复扣费、重复生成。
6. 合并导出时，分辨率、编码、音轨、帧率不一致导致 concat 失败。
7. Worker 崩溃后任务状态卡死。
8. 平台链接下载涉及版权、授权、反爬、SSRF 和平台协议风险。

所以成熟方案的核心是：**状态机 + 队列 + 幂等 + 事务账本 + 资产规范化 + AI 输出校验 + 失败补偿 + 可观测性**。

# 二、推荐技术架构

## 1. 服务分层

建议采用下面这套架构：

```text
Frontend
  Next.js / React
  ├─ 上传页
  ├─ 解析进度页
  ├─ 分镜编辑页
  ├─ 生成进度页
  └─ 导出下载页

API Service
  Node.js / NestJS 或 Next.js Route Handler
  ├─ Project API
  ├─ Shot / ShotGroup API
  ├─ Generate API
  ├─ Export API
  ├─ Credit API
  └─ Webhook / SSE / Polling API

Database
  PostgreSQL + Prisma
  ├─ Project
  ├─ Asset
  ├─ Frame
  ├─ Shot
  ├─ ShotGroup
  ├─ ProviderTask
  ├─ JobRecord
  ├─ CreditLedger
  └─ ExportRecord

Queue
  Redis + BullMQ
  ├─ download-video
  ├─ parse-video
  ├─ ai-parse
  ├─ slice-audio
  ├─ generate-video
  ├─ poll-provider-task
  └─ merge-video

Storage
  OSS / S3 Compatible Storage + CDN
  ├─ source video
  ├─ normalized video
  ├─ extracted frames
  ├─ audio slices
  ├─ generated group videos
  └─ exported final video

Workers
  Dockerized Worker Pool
  ├─ FFmpeg / ffprobe
  ├─ Gemini adapter
  ├─ Seedance adapter
  ├─ Media normalize worker
  └─ Merge worker
```

BullMQ 适合这类 Node.js 异步任务系统，它基于 Redis，支持横向扩展、并发、失败重试、延迟任务、崩溃恢复等能力。([BullMQ][1])

**注意点：**

* API 服务不要直接跑 FFmpeg、下载视频、轮询 AI 任务。
* 所有耗时任务必须进队列。
* 每个队列任务必须可重试、可幂等、可恢复。
* Worker 和 API 要分开部署，避免 FFmpeg 抢占 Web 服务 CPU。
* 所有外部 AI 调用都要通过 Provider Adapter 封装，不要把 Seedance/Gemini 代码散落在业务里。

# 五、Phase 1：视频导入落地方案

## 上传流程

```text
POST /api/projects
  → 创建 Project(status=CREATED)
  → 返回 projectId + presignedUploadUrl

前端直传 OSS
  → 上传完成后 POST /api/projects/:id/confirm-upload
  → Project.status = PARSING
  → 入队 parse-video
```

## 平台链接流程

```text
POST /api/projects
  body: { sourceType: "PLATFORM_URL", sourceUrl }
  → Project.status = DOWNLOADING
  → 入队 download-video
```

`download-video worker`：

```text
1. 校验 URL 白名单
2. 防 SSRF 校验
3. 下载到临时目录
4. ffprobe 检查格式、时长、大小、分辨率
5. 上传 source video 到 OSS
6. 转码 normalized video
7. 入队 parse-video
```

**注意点：**

* 平台链接下载必须处理版权、平台服务条款、用户授权。不要绕过 DRM，不要做反爬破解。
* 必须做 SSRF 防护：禁止访问内网 IP、localhost、云元数据地址、重定向到内网。
* 限制视频大小、时长、分辨率。建议 MVP：单视频不超过 3 分钟、500MB；商业版按套餐放开。
* 上传视频后先做 `ffprobe`，不合规直接失败，不要进入 AI 流程烧钱。
* iPhone 视频常见可变帧率、HEVC、旋转元数据，必须先 normalize。
* 所有下载、转码临时文件要放到独立目录，任务结束清理。

# 六、Phase 2：视频解析落地方案

## 1. 元数据获取

```bash
ffprobe -v error \
  -select_streams v:0 \
  -show_entries stream=width,height,r_frame_rate,avg_frame_rate,codec_name,duration \
  -show_entries format=duration,size,bit_rate \
  -of json input.mp4
```

记录：

```ts
durationMs
width
height
fps
codec
hasAudio
rotation
bitrate
```

**注意点：**

* `r_frame_rate` 和 `avg_frame_rate` 可能不一致，导出时建议统一成固定 fps。
* `duration` 要以 format 和 stream 双重校验。
* 有些视频无音轨，后续切音频必须生成静音音频或标记 `hasAudio=false`。

---

## 2. 视频标准化

建议所有原始视频先转标准中间文件：

```bash
ffmpeg -y -i input.mp4 \
  -vf "scale='min(1280,iw)':-2,fps=24,format=yuv420p" \
  -c:v libx264 -preset veryfast -crf 23 \
  -c:a aac -b:a 128k \
  -movflags +faststart \
  normalized.mp4
```

**注意点：**

* 后续抽帧、切音频、合并都基于 normalized video。
* 这样可以规避 HEVC、可变帧率、旋转元数据、奇怪音频编码造成的随机失败。
* 如果业务要求高保真，可以保留 source video，同时用 normalized video 做处理。

---

## 3. 抽帧策略

你原来的策略是：**场景检测 + 长镜头内均匀补帧，总帧数 ≤20，每帧带 timestamp 和 isSceneChange 标记**。这个方向是对的。

建议升级为生产策略：

```text
输入：normalized video

A. 场景切换检测
   - scene threshold 默认 0.32
   - 得到 sceneChangeTimestamps

B. 基础采样
   - 每个 scene 起点取 1 帧
   - 长 scene 每 2~4 秒补 1 帧

C. 黑帧/模糊帧过滤
   - 如果帧过暗、过亮、运动模糊严重，向后偏移 200~500ms 重取

D. 总数控制
   - 短视频 ≤20 帧
   - 长视频不要固定 20 帧，应按时长动态上限
   - 例如 maxFrames = clamp(20, ceil(durationSec / 3), 80)

E. 帧上传 OSS
   - 生成 frame records
```

示例命令：

```bash
ffmpeg -i normalized.mp4 \
  -vf "select='gt(scene,0.32)',showinfo" \
  -vsync vfr \
  scene_%05d.jpg
```

**注意点：**

* `threshold=0.32` 不能写死，建议存到配置表。
* 动漫、快剪、闪白视频 threshold 要升高，否则切太碎。
* 慢镜头、长镜头、固定机位视频 threshold 要降低或依赖均匀补帧。
* 只抽 20 帧适合短视频；如果用户上传 2~3 分钟视频，20 帧会漏掉大量内容。
* Gemini 的视频理解默认视觉采样为 1 FPS，快速运动或快速切换场景可能漏细节，所以你自己做抽帧并携带精准时间戳是合理的。([Google AI for Developers][2])

## 4. Gemini 解析方案

建议不要只让 Gemini “自由发挥”，必须要求结构化 JSON 输出。

### 输入

```json
{
  "videoDurationMs": 38120,
  "frames": [
    {
      "timestampMs": 0,
      "url": "https://...",
      "isSceneChange": true
    },
    {
      "timestampMs": 4200,
      "url": "https://...",
      "isSceneChange": true
    }
  ],
  "requirements": {
    "language": "zh-CN",
    "output": "ParsedShot[]",
    "promptFor": "Seedance image-to-video",
    "maxPromptLength": 500
  }
}
```

### 输出 Schema

```ts
const ParsedShotSchema = z.object({
  orderIndex: z.number().int(),
  startTimeMs: z.number().int(),
  endTimeMs: z.number().int(),
  scene: z.string(),
  shotType: z.string().optional(),
  cameraMove: z.string().optional(),
  characters: z.array(z.object({
    name: z.string().optional(),
    appearance: z.string(),
    clothing: z.string().optional(),
    action: z.string().optional()
  })).default([]),
  hasFace: z.boolean(),
  suggestedPrompt: z.string().max(500),
  confidence: z.number().min(0).max(1).optional()
})
```

**注意点：**

* Gemini 可以基于视频中的时间戳进行提问和分析，官方文档说明可以使用 `MM:SS` 形式引用视频时间点。([Google AI for Developers][2])
* AI 返回必须走 Zod/JSON Schema 校验。
* JSON 解析失败要做一次 repair retry，不要直接失败。
* 如果 AI 输出 shot 时间重叠、倒序、超出视频总时长，要自动修正或进入人工确认状态。
* `suggestedPrompt` 要面向 Seedance，不要写成“画面中有一个人”，而要写成“人物做什么动作，镜头如何运动，画面如何变化”。
* 建议加一个二次 prompt rewrite 步骤，把 Gemini 的描述转成稳定的生成指令。

## 5. Shot 边界生成策略

不要完全依赖 AI 给的分镜边界。建议：

```text
主边界：FFmpeg scene detection timestamps
辅助边界：Gemini 对画面内容的理解
修正规则：
  - shot.startTimeMs 必须来自 scene boundary 或补帧点附近
  - shot.endTimeMs = 下一个 shot.startTimeMs
  - 最后一个 shot.endTimeMs = videoDurationMs
```

**注意点：**

* AI 很容易把动作描述对，但时间边界写错。
* 所有 shot 必须满足：

  * `startTimeMs >= 0`
  * `endTimeMs <= durationMs`
  * `endTimeMs > startTimeMs`
  * `orderIndex` 连续
  * 时间线无重叠、无空洞，除非明确允许跳剪。

## 6. groupShots 生产策略

你当前设计：**每个 Shot 独立成 1 个 ShotGroup，<4s 的 Shot 向前合并**。这个是非常适合落地的，因为 prompt 短、首帧准、生成失败影响面小。

建议升级成配置化：

```ts
type GroupConfig = {
  minDurationSec: number     // 例如 4
  maxDurationSec: number     // 例如 15
  maxPromptChars: number     // 例如 500
  mergeShortShotStrategy: 'PREVIOUS' | 'NEXT' | 'SIMILAR_SCENE'
}
```

伪代码：

```ts
function groupShots(shots, config) {
  const groups = []

  for (const shot of shots) {
    if (shot.durationSec >= config.minDurationSec) {
      groups.push([shot])
      continue
    }

    const prev = groups[groups.length - 1]

    if (prev && totalDuration(prev) + shot.durationSec <= config.maxDurationSec) {
      prev.push(shot)
    } else {
      groups.push([shot])
    }
  }

  return groups.map((shots, index) => ({
    groupIndex: index,
    startTimeMs: shots[0].startTimeMs,
    endTimeMs: shots[shots.length - 1].endTimeMs,
    durationMs: sum(shots),
    firstFrameUrl: shots[0].coverUrl,
    finalScript: mergePrompt(shots)
  }))
}
```

**注意点：**

* “向前合并”不一定总是最佳。第一个 shot 如果小于 4s，只能向后合并。
* 如果短 shot 和前一个 shot 场景完全不同，向前合并会让首帧和 prompt 不一致。
* 更成熟的策略是：优先合并到同场景/同人物/同动作连续的相邻 shot。
* 合并后如果 prompt 超过模型限制，需要重写为单段运动描述，而不是简单拼接。
* 生成时长要 `clamp(group.durationSec, minDurationSec, maxDurationSec)`。

---

## 7. 音频切片

```bash
ffmpeg -y -i normalized.mp4 \
  -ss 12.300 -to 18.800 \
  -vn \
  -ac 2 -ar 44100 \
  -c:a libmp3lame -b:a 128k \
  group_003.mp3
```

**注意点：**

* 无音频视频要生成静音音频，或者 `referenceAudioUrl=null`。
* 切音频时要避免 `-ss` 放错位置导致时间不准。高精度切片建议 `-i input -ss start -to end`。
* 后续导出要明确音频策略：

  * `ORIGINAL_MASTER`：最终视频使用原始音频，推荐默认。
  * `GENERATED_AUDIO`：使用 Seedance 生成视频自带音频。
  * `MIXED`：生成音频 + 原始音频混合。
  * `MUTED`：无音频导出。
* 如果目标是“复刻原视频节奏”，建议最终导出默认使用原始音频作为主音轨，而不是完全依赖每段生成视频的音频。

# 八、Phase 4：单组生成落地方案

## API 设计

```http
POST /api/shot-groups/:id/generate
Headers:
  Idempotency-Key: generate:{groupId}:{scriptHash}

Body:
{
  "resolution": "480p",
  "model": "seedance-2.0-fast",

}
```

返回：

```json
{
  "shotGroupId": "sg_123",
  "status": "QUEUED",
  "jobId": "generate:sg_123:hash_xxx"
}
```

## 后端处理流程

```text
1. 校验用户权限
2. 读取 ShotGroup + Shots
3. 生成 finalScript
   - timelineScript 非空：用用户编辑稿
   - 否则：用 shot.prompt 或合并重写
4. 校验 prompt 长度
5. 计算 scriptHash
6. 幂等检查
   - 同 scriptHash 已成功：直接返回成功
   - 同 scriptHash 正在生成：返回当前任务
7. 事务内冻结积分
8. 更新 ShotGroup.genStatus = QUEUED
9. 入队 generate-video
```

**注意点：**

* 第 6 步必须做，否则用户双击会重复生成。
* 第 7 步必须和任务入队设计成强一致或可补偿。
* 如果入队失败，积分 HOLD 必须释放。
* 如果任务已成功但用户没刷新页面，再点生成应该直接返回已有结果，不要再扣费。

## generate-video worker 流程

```text
1. 加分布式锁 lock:shotGroup:{id}
2. 再次读取 DB，确认状态仍需生成
3. 生成 provider request
4. 创建 ProviderTask 记录
5. 调 Seedance create task
6. 保存 providerTaskId
7. 入队 poll-provider-task
```

请求结构建议抽象为：

```ts
type VideoGenerationRequest = {
  provider: 'SEEDANCE'
  model: string
  prompt: string
  firstFrameUrl: string
  referenceAudioUrl?: string
  durationSec: number
  resolution: '480p' | '720p' | '1080p'
  aspectRatio: string
  seed?: number
  safetyIdentifier: string
}
```

Seedance 2.0 的官方 prompt 指南强调，除文本外，也可以提供图片、音频、视频等素材用于参考控制，并在 prompt 中明确说明“使用图 1 的构图”或“匹配视频 2 的运动”等参考对象。([BytePlus][6])

**注意点：**

* first frame URL 必须是外部服务可访问的 URL。若 OSS 是私有桶，需要生成足够长的 signed URL。
* signed URL 有效期要覆盖 Seedance 拉取素材的时间，建议至少 2 小时。
* 生成服务返回的是异步任务，不能在 HTTP 请求里等待完成。
* ProviderTask 创建后，即使 Worker 崩溃，也能靠 providerTaskId 继续查询。
* AI 服务可能限流，队列要配置并发和速率限制。
* 生成失败不要直接 `FAILED` 后结束，要区分：

  * 参数错误：不可重试。
  * 429/限流：延迟重试。
  * 5xx：指数退避重试。
  * 内容安全失败：不可重试，提示用户修改 prompt 或首帧。
  * 超时未知：进入 `UNKNOWN` 或继续轮询，不要贸然扣费或退款。

## poll-provider-task worker

```text
1. 查询 ProviderTask
2. 调 Seedance retrieve task
3. 如果 running：
   - 更新 lastPollAt
   - 计算 nextPollAt
   - 重新入 delayed queue
4. 如果 succeeded：
   - 下载 provider output video
   - 上传到 OSS
   - 写 Asset
   - 更新 ShotGroup.genVideoUrl
   - 更新 ShotGroup.genStatus = SUCCEEDED
   - 更新关联 Shots
   - CreditLedger: COMMIT
5. 如果 failed：
   - ShotGroup.genStatus = FAILED
   - CreditLedger: RELEASE
```

**注意点：**

* 第三方输出视频必须下载回自己的 OSS，不要直接保存第三方临时 URL。
* 轮询间隔不要固定 1 秒，建议：

  * 前 30 秒：每 5 秒
  * 30 秒后：每 10~15 秒
  * 超过最大等待时间：标记 TIMEOUT，但保留后台 reconcile 能力。
* `COMMIT` 和 `ShotGroup.genStatus=SUCCEEDED` 必须在同一个数据库事务内完成。
* 如果上传 OSS 成功但 DB 更新失败，要有补偿扫描任务。
* 如果 DB 成功但积分 commit 失败，要有账本对账任务。

# 九、Phase 5：合并导出落地方案

你当前方案是：所有 ShotGroup 成功后，按 groupIndex 排序，用 FFmpeg concat 拼接。方向正确。

但生产环境里，直接 concat 很容易失败，因为不同生成片段可能存在：

* 分辨率不同。
* fps 不同。
* codec 不同。
* timebase 不同。
* 有的有音轨，有的无音轨。
* 音频采样率不同。
* 色彩格式不同。

FFmpeg concat demuxer 要求待拼接文件具有相同 stream，例如 codec、time base 等要一致。([FFmpeg Trac][7])

所以成熟方案必须先做 normalize。

## 导出流程

```text
POST /api/projects/:id/export

1. 校验权限
2. 校验所有 ShotGroup.genStatus = SUCCEEDED
3. 校验所有 generated video 已回存 OSS
4. 创建 ExportRecord(status=QUEUED)
5. 入队 merge-video
```

## merge-video worker

```text
1. 下载所有 group video 到临时目录
2. 对每个 group video 做 normalize
3. 生成 concat list
4. FFmpeg concat
5. 根据 audioMode 处理音轨
6. 输出 final mp4
7. 上传 OSS
8. 写 Asset(type=EXPORTED_VIDEO)
9. Project.status = EXPORTED
10. ExportRecord.status = SUCCEEDED
```

## 每段视频 normalize

```bash
ffmpeg -y -i group_001.mp4 \
  -vf "scale=854:480:force_original_aspect_ratio=decrease,pad=854:480:(ow-iw)/2:(oh-ih)/2,fps=24,format=yuv420p" \
  -c:v libx264 -preset veryfast -crf 20 \
  -c:a aac -ar 44100 -ac 2 -b:a 128k \
  -movflags +faststart \
  normalized_group_001.mp4
```

如果某段没有音轨，补静音：

```bash
ffmpeg -y -i group_001.mp4 \
  -f lavfi -i anullsrc=channel_layout=stereo:sample_rate=44100 \
  -shortest \
  -c:v libx264 -c:a aac \
  normalized_group_001.mp4
```

## concat

```text
file 'normalized_group_001.mp4'
file 'normalized_group_002.mp4'
file 'normalized_group_003.mp4'
file 'normalized_group_004.mp4'
```

```bash
ffmpeg -y -f concat -safe 0 -i concat.txt \
  -c copy \
  merged.mp4
```

如果 concat copy 失败，则回退到重编码 concat：

```bash
ffmpeg -y -f concat -safe 0 -i concat.txt \
  -c:v libx264 -preset veryfast -crf 20 \
  -c:a aac -b:a 128k \
  -movflags +faststart \
  merged.mp4
```

**注意点：**

* 不要直接拼第三方返回视频。
* normalize 后再 concat，成功率高很多。
* 导出不应该再次扣生成积分，最多收导出处理费。
* 导出失败可以无限重试，不应要求用户重新生成。
* 导出成功后，下载链接建议是 signed URL，不建议永久公开。

# 十三、幂等与失败补偿

这是上线成败的关键。

## 必须幂等的操作

| 操作           | 幂等 Key                                                            |
| ------------ | ----------------------------------------------------------------- |
| 创建 Project   | `userId + sourceHash/uploadSessionId`                             |
| 视频下载         | `projectId + sourceUrlHash`                                       |
| 视频 normalize | `sourceAssetChecksum`                                             |
| 抽帧           | `projectId + normalizedVideoChecksum + parseConfigHash`           |
| AI 解析        | `projectId + frameSetHash + promptVersion`                        |
| 单组生成         | `shotGroupId + scriptHash + modelVersion + duration + resolution` |
| 积分冻结         | `generate:${shotGroupId}:${scriptHash}:hold`                      |
| 积分扣除         | `generate:${shotGroupId}:${scriptHash}:commit`                    |
| 导出           | `projectId + groupVideoHashList + exportConfigHash`               |

## 补偿任务

每天或每小时跑 reconcile：

```text
1. 找 Project.status=PARSING 但超过 30 分钟无更新
2. 找 ShotGroup.status=RUNNING 但 ProviderTask 无轮询
3. 找 ProviderTask=SUCCEEDED 但 genAsset 未写入
4. 找 Credit HOLD 超过 2 小时但无 ProviderTask
5. 找 EXPORTED 但 Asset 不存在
6. 找第三方 URL 未回存 OSS 的记录
```

**注意点：**

* Worker 崩溃是常态，不是异常。
* 任何状态卡住都应该能被扫描任务恢复。
* 不要让用户联系客服才能解锁项目。

# 十四、Prompt 生成规范

你的方向是“不要再拼接长 prompt，直接用 Gemini 的 suggestedPrompt”。这个适合单 shot 组。

但如果短 shot 合并成组，建议走一次 prompt rewrite：

## 单 Shot Prompt 模板

```text
主体：{人物/物体}
动作：{连续动作}
场景：{环境}
镜头：{景别 + 运镜}
风格：{画质/光影/氛围}
约束：保持首帧人物外貌、服装、构图一致；动作自然连续；不要突然切换场景。
```

## 示例

```text
白衣男子盘腿坐在山巅石台上，衣袍随风轻微飘动，镜头从正面缓慢推进到中近景，背景云海翻涌，金色晨光照亮人物轮廓。保持首帧人物外貌、服装和构图一致，动作自然，画面电影感。
```

## 多 Shot 合并 Prompt

```text
延续首帧画面中的白衣男子，他先盘腿凝神，随后周围气流增强，衣袍和长发被风吹起，镜头缓慢环绕并推进，最后人物睁眼释放金色光芒。保持人物外貌、服装、场景连续，不要切换到无关环境。
```

**注意点：**

* Prompt 要写“运动过程”，不是静态描述。
* 首帧已经提供构图，不要在 prompt 里写过多互相冲突的构图描述。
* 人脸一致性要求要明确写。
* 避免抽象词堆叠，例如“史诗感、震撼、大片级”太多会降低可控性。
* 中文 prompt 长度建议内部限制，比如 300~500 字以内。
* Prompt 修改后必须更新 `scriptHash`，旧结果标记为 stale。

# 十五、首帧策略

你现在用 `shots[0].coverUrl` 作为 first frame，很正确。

生产上建议增强：

```text
firstFrame 选择优先级：
1. shot.startTimeMs + 200ms 的清晰帧
2. 如果是黑帧/模糊帧，向后找 500ms
3. 如果仍然不可用，使用场景内亮度和清晰度评分最高的帧
4. 用户可手动更换首帧
```

**注意点：**

* 首帧决定生成视频的构图和人物一致性。
* 转场瞬间的帧不要作为首帧。
* 有人脸时要优先选脸部清晰、无遮挡的帧。
* `hasFace=true` 的 ShotGroup 可以走更严格的内容安全和隐私策略。

# 十六、积分与计费方案

## 推荐计费时机

```text
用户点击生成
  → 预估费用
  → 冻结积分 HOLD
  → 生成成功 COMMIT
  → 生成失败 RELEASE
```

## 费用计算

```ts
cost = baseCost[model][resolution] * durationSec * multiplier
```

示例：

```ts
const estimatedCredits = calculateCredits({
  model,
  resolution,
  durationSec,
  hasReferenceAudio,
  priority

```

**注意点：**

* 不要按用户传入的 duration 计费，要按最终送给模型的 `genDurationSec`。
* 失败释放积分，内容安全失败是否收费要看你的产品规则，但必须提前写清楚。
* 第三方返回成功但你下载失败，不应该让用户承担损失，应重试下载或人工补偿。
* 账本只能追加记录，不能直接改余额字段了事。
* 需要后台对账：第三方消耗、平台扣费、用户账本三者要对得上。

# 十七、安全与合规

## 上传安全

* 校验 MIME 和文件 magic number。
* 限制大小、时长、分辨率。
* 病毒扫描可以第二阶段接入。
* 私有 OSS bucket + signed URL。
* 不要让用户上传后直接变成永久公网 URL。

## 平台链接安全

* URL 白名单。
* 禁止内网 IP。
* 禁止重定向到内网。
* 禁止访问云厂商 metadata 地址。
* 设置下载超时、最大响应体、最大重定向次数。
* 不支持 DRM、付费、未授权内容下载。

## 内容安全

* Prompt 审核。
* 首帧/原视频内容审核。
* 生成结果审核。
* 人脸、未成年人、名人、版权角色要做风险标记。

## 密钥安全

* Gemini、Seedance、OSS 密钥必须放 Secret Manager。
* 不要写进 `.env` 后提交仓库。
* Provider request/response 日志要脱敏。

**注意点：**

* AI 生成业务最容易踩的是版权、肖像权、平台内容抓取。
* 用户粘贴平台链接时，产品层面最好加确认：“我确认拥有该视频的使用权或已获得授权。”
* 如果后续商用，建议增加内容投诉和下架机制。

# 十八、可观测性与后台运营

## 必须打点的指标

| 指标                                  | 价值        |
| ----------------------------------- | --------- |
| project_parse_success_rate          | 解析成功率     |


**注意点：**

* 没有 traceId，线上排查会非常痛苦。

* 要能看到第三方原始错误码，而不是只显示“生成失败”。

# 十九、部署方案

## MVP 部署

```text
1 台 API 服务
1 台 Worker 服务
1 个 PostgreSQL
1 个 Redis
1 个 OSS bucket
```

适合内测和小流量。

## 生产部署

```text
API Service
  2+ replicas

Worker Pool
  download workers
  parse/ffmpeg workers
  ai/generate workers
  merge workers

Redis
  managed Redis

PostgreSQL
  managed PostgreSQL + backup + PITR

Storage
  private OSS bucket + CDN

Observability
  Sentry + Prometheus + Grafana
```

**注意点：**

* FFmpeg Worker 最好独立机器或独立容器组，CPU 配额要高。
* Generate Worker 不吃 CPU，但吃第三方 API 限流，适合轻量容器。
* Merge Worker 磁盘 IO 高，临时目录要足够大。
* 所有 Worker 容器镜像内固定 FFmpeg 版本，避免不同机器输出不一致。
* 定期清理临时文件和过期中间资产。

# 二十二、最关键的注意点汇总

| 位置          | 必须注意                                             |
| ----------- | ------------------------------------------------ |
| 视频导入        | 平台链接要做授权、SSRF、防 DRM 风险                           |
| 视频处理        | 原视频必须 normalize 后再抽帧、切音频                         |
| 抽帧          | scene threshold 要配置化，不能固定 0.32                   |
| Gemini      | 必须结构化 JSON 输出 + Schema 校验                        |
| Shot 边界     | FFmpeg 时间戳为主，AI 描述为辅                             |
| ShotGroup   | `<4s` 合并规则要配置化，首个短 shot 要能向后合并                   |
| Prompt      | 单 shot 不拼长 prompt，多 shot 合并要 rewrite             |
| 首帧          | 不要取黑帧/转场帧，建议 start + 200ms                       |
| 生成          | 所有生成请求必须幂等                                       |
| 积分          | 先冻结，成功扣除，失败释放                                    |
| Seedance 结果 | 第三方视频 URL 必须立即回存 OSS                             |
| 导出          | 所有片段先统一编码、fps、分辨率、音轨，再 concat                    |
| 状态          | DB 是唯一事实源，不能依赖队列状态                               |
| Worker      | 所有任务可重试、可恢复、可补偿                                  |
| 监控          | 必须有 projectId、shotGroupId、providerTaskId、traceId |
| 成本          | 控制帧数、时长、并发、重试次数和模型档位                             |

# 最终落地结论

你的业务全景图方向是正确的，但要从“流程图”升级为“生产级系统”，需要补上这几层：

```text
业务流程层：
  视频导入 → 解析 → 编辑 → 生成 → 导出

工程可靠性层：
  状态机 → 队列 → 幂等 → 重试 → 补偿 → 账本

媒体处理层：
  normalize → 抽帧 → 切音频 → 片段标准化 → concat

AI 稳定性层：
  Gemini schema 校验 → Prompt rewrite → Seedance provider task → 结果回存

运营安全层：
  权限 → 存储安全 → 内容安全 → 版权授权 → 日志监控 → 成本控制
```

按照这个方案做，第一版就可以上线内测；后续主要迭代点是**手动拆分/合并、首帧替换、多版本生成、成本看板和内容审核**。
